# Software Specification Document (SSD)

## 1. Мета документа

Цей документ описує, як саме система TaskManager реалізує бізнес-вимоги з BRD на рівні архітектури та системних можливостей. Тут зафіксовані підсистеми, потоки даних, правила доступу, інтеграції, інтерфейси та нефункціональні обмеження. Документ не опускається до коду окремих класів або SQL-реалізації.

## 2. Системна архітектура

TaskManager побудований як клієнт-серверний вебзастосунок із розділеною архітектурою.

* **Frontend**: React SPA з компонентною структурою, клієнтською маршрутизацією та централізованим сховищем стану.
* **Backend**: PHP MVC-застосунок з єдиною точкою входу, маршрутизатором, контролерами, middleware та моделями.
* **Data layer**: реляційна база даних із зовнішніми ключами та правилами очищення пов'язаних записів.
* **Integrations**: SMTP-пошта для запрошень до команди.
* **Transport**: JSON API поверх HTTPS.
* **Authentication**: короткоживучий токен доступу та довгоживучий refresh token, який зберігається в захищеному cookie.

## 3. Як системні механізми підтримують бізнес-цілі

| Бізнес-ціль | Системний механізм |
| --- | --- |
| Швидка робота без перезавантажень | SPA-інтерфейс, локальне оновлення стану та модальні вікна для створення й редагування |
| Прозорість виконання роботи | Дошки, списки, статуси, пріоритети та візуальне відображення виконавців |
| Організованість простору | Персональні дашборди, теги та фільтрація задач за контекстом |
| Командна взаємодія | Команди, запрошення, членство, спільні дошки та призначення виконавців |
| Безпека та приватність | Auth middleware, перевірка прав доступу, хешовані паролі, токени доступу та HTTPS |
| Збереження сесії | Refresh token у cookie та автоматичне поновлення access token |

## 4. Підсистеми

### 4.1. Ідентифікація, автентифікація та профіль

Ця підсистема забезпечує:

* реєстрацію нового користувача;
* вхід у систему за email і паролем;
* оновлення access token без повторного входу;
* вихід із системи;
* зміну email і пароля;
* видалення облікового запису;
* завантаження та оновлення аватара.

Ключові правила:

* email повинен бути унікальним;
* пароль зберігається тільки у хешованому вигляді;
* зміну чутливих даних слід підтверджувати поточним паролем;
* усі приватні дії виконуються лише після перевірки access token.

### 4.2. Особистий робочий простір

Ця підсистема відповідає за індивідуальні дашборди та особисті списки задач.

Система повинна підтримувати:

* створення, оновлення та видалення персональних дашбордів;
* створення, редагування, видалення та впорядкування списків;
* створення задач у межах особистого простору;
* редагування, видалення та переміщення задач між списками;
* відкриття деталей задачі в модальному вікні без втрати контексту сторінки.

### 4.3. Керування задачами

Задача є основною одиницею роботи в системі. Вона може містити:

* назву;
* опис;
* час початку;
* дедлайн;
* параметр нагадування або сповіщення;
* пріоритет;
* статус виконання;
* опціональний зв'язок із батьківською задачею для ієрархії підзадач.

Система повинна забезпечити:

* створення задач у відповідному списку;
* оновлення дозволених полів без повторного завантаження сторінки;
* переміщення задач між списками;
* зміну порядку задач у списку;
* пошук задач за назвою в межах доступного користувачу простору;
* каскадне очищення зв'язків під час видалення задачі.

### 4.4. Керування тегами

Теги використовуються для класифікації та швидкої фільтрації задач.

Система повинна дозволяти:

* створювати, редагувати та видаляти кольорові теги;
* зберігати теги окремо для кожного користувача;
* не допускати дублювання імен тегів у межах одного користувача;
* прикріплювати та від'єднувати теги від задачі;
* фільтрувати задачі за вибраним тегом.

### 4.5. Команди та запрошення

Ця підсистема реалізує групову роботу над спільним проєктом.

Система повинна підтримувати:

* створення команди;
* автоматичне призначення автора команди власником/адміністратором;
* перегляд складу команди та активних запрошень;
* надсилання запрошення на email;
* перевірку статусу запрошення;
* прийняття або скасування запрошення;
* відкликання запрошення власником команди;
* видалення учасника з команди власником.

Запрошення мають бути обмежені часом дії та пов'язані з конкретним користувачем і конкретною командою.

### 4.6. Командні дошки та призначення задач

Командний простір працює за тією ж логікою, що й особистий, але доступ до нього мають лише учасники команди.

Система повинна забезпечити:

* перегляд командних дошок і їхніх списків;
* створення, оновлення та видалення списків у межах команди;
* створення, оновлення, переміщення, впорядкування та видалення командних задач;
* призначення задачі одному або кільком учасникам команди;
* перегляд списку призначень для конкретної задачі;
* зняття призначення без видалення самої задачі.

### 4.7. Пошук

Підсистема пошуку допомагає швидко знаходити задачі за назвою.

Пошук повинен:

* працювати в межах доступних користувачу особистих і командних просторів;
* повертати лише ті записи, до яких у користувача є доступ;
* віддавати результат у формі JSON для подальшого фільтрування на клієнті.

## 5. Основні сценарії взаємодії

### 5.1. Реєстрація та вхід

1. Користувач надсилає email, пароль та інші обов'язкові дані.
2. Сервер перевіряє коректність і унікальність email.
3. Пароль хешується та зберігається разом із сіллю.
4. Система повертає access token і refresh token.
5. Frontend зберігає refresh token у cookie та використовує access token для приватних запитів.

### 5.2. Оновлення сесії

1. Якщо access token протерміновано, клієнт бере refresh token із cookie.
2. Клієнт надсилає запит на відновлення сесії.
3. Сервер перевіряє refresh token, термін його дії та зв'язок із користувачем.
4. Якщо перевірка успішна, повертається новий access token.
5. Користувач продовжує роботу без повного перезавантаження застосунку.

### 5.3. Робота із задачами

1. Користувач відкриває потрібну дошку або список.
2. Створення та редагування задачі відбувається в модальному вікні.
3. Після збереження клієнт оновлює лише відповідну частину інтерфейсу.
4. Переміщення між списками та зміна порядку виконуються без втрати контексту дошки.

### 5.4. Командна взаємодія

1. Власник створює команду та запрошує учасника за email.
2. Система надсилає повідомлення із запрошенням і токенізованим посиланням.
3. Запрошений користувач перевіряє запрошення та приймає його.
4. Після підтвердження він отримує доступ до командного простору.
5. Власник може призначати задачі учасникам і контролювати склад команди.

## 6. Модель даних і цілісність

Система повинна оперувати такими ключовими сутностями:

* **Користувач** — профіль користувача, аватар, email, пароль та статус сесії.
* **Токен авторизації** — довготривала сесійна сутність для відновлення доступу.
* **Завдання** — окрема одиниця роботи з полями опису, статусу, пріоритету та дедлайну.
* **Тег** — кольорова мітка користувача.
* **Дошка завдань** — особистий або командний простір організації задач.
* **Список завдань** — колонка або список усередині дошки.
* **Команда** — груповий простір співпраці.
* **Запрошення в команду** — активне або минуле запрошення до команди.

Правила цілісності даних:

* видалення задачі має очищати пов'язані призначення, теги та зв'язки зі списками;
* видалення тегу має прибирати всі зв'язки з задачами;
* видалення команди має очищати запрошення, членство, призначення та пов'язані командні простори;
* видалення користувача має прибирати його особисті дані та пов'язані записи, щоб не утворювалися «сирітські» дані;
* доступ до персональних і командних сутностей повинен перевірятися на сервері перед кожною зміною.

## 7. Інтерфейси та обмін даними

### 7.1. HTTP та JSON

Усі взаємодії між клієнтом і сервером повинні відбуватися через JSON API.

* запити надсилаються з `Content-Type: application/json`;
* відповіді повертаються у JSON-форматі;
* для приватних дій використовується заголовок `Authorization: Bearer <token>`.

### 7.2. Сесії та cookie

* refresh token зберігається в захищеному cookie;
* cookie має використовувати захищені параметри для браузера;
* якщо продукт використовує cookie для підтримки сесії, інтерфейс повинен інформувати користувача про це на рівні політики використання даних.

### 7.3. Електронна пошта

* запрошення до команди надсилаються через SMTP;
* лист містить токенізоване посилання для прийняття запрошення;
* термін дії запрошення обмежений.

### 7.4. Клієнтський інтерфейс

* сторінки повинні працювати як SPA без повного перезавантаження;
* форми створення, редагування та перегляду задач мають відкриватися в модальних вікнах;
* основні області інтерфейсу повинні включати верхню панель, бічну навігацію та центральну робочу зону;
* мобільна та десктопна версії повинні зберігати зручність роботи з дошками і списками.

## 8. Нефункціональні вимоги

### 8.1. Продуктивність

* Час реакції сервера для типової операції не повинен перевищувати 400 мс за нормальних умов роботи.
* Клієнт не повинен перезавантажувати сторінку після кожної дії.
* Оновлення має бути локальним і стосуватися лише змінених частин інтерфейсу.

### 8.2. Безпека

* Паролі мають зберігатися тільки у хешованому вигляді з використанням сучасного алгоритму та солі.
* Приватні ендпоінти повинні бути захищені access token'ом.
* Refresh token має зберігатися окремо від access token і не передаватися відкрито в URL.
* Усі операції над чужими даними повинні блокуватися на сервері, а не тільки на клієнті.
* Усі мережеві запити мають виконуватися через HTTPS.

### 8.3. Надійність та цілісність

* Помилки валідації, авторизації та бізнес-правил мають повертатися у стандартизованому JSON-форматі з відповідним HTTP-кодом.
* Пов'язані записи повинні очищатися синхронно під час видалення сутностей.
* Термін дії запрошень до команди та refresh token'ів має контролюватися системою.

### 8.4. Зручність використання

* Інтерфейс повинен бути адаптивним для мобільних, планшетів і десктопів.
* Користувач повинен бачити основну інформацію про задачу без переходу на іншу сторінку.
* Критичні дії повинні мати зрозумілі підтвердження та помилки.

## 9. Обмеження та припущення

* Backend реалізується на чистому PHP у стилі MVC без великого фреймворка.
* Frontend реалізується як React SPA.
* Для зберігання даних використовується реляційна база даних.
* Серверне середовище повинно підтримувати маршрутизацію через єдину точку входу та переписування URL.
* Для роботи запрошень і сесій необхідна доступна поштова інфраструктура та підтримка cookie у браузері.
* Проєкт орієнтований на сучасні браузери, які підтримують JavaScript і захищений обмін даними.
